# Vocabulary Feature — Design Document

**Project:** English Tutor Telegram Bot (via OpenClaw)
**User:** Mom (beginner, Russian-speaking adult)
**Date:** February 2026

---

## 1. Overview

The vocabulary feature is the core learning module of the English tutor bot. It manages word acquisition through a cycle of exposure, testing, and spaced review. The bot runs 3 sessions per day (~7-10 minutes each), all following the same structure. The system is designed so session composition can be easily swapped per time-of-day or day-of-week in the future.

---

## 2. Session Structure

Every session runs an ordered array of Exercise instances, built by a `build_session()` function and executed sequentially by a `SessionExecutor`.

**MVP session pipeline (identical every time):**

1. **VocabExercise** — show 5 words with translations, then test 5 words
2. **GrammarExercise** — find the mistake in a sentence
3. **VoiceExercise** — listen to a phrase and translate

Mini-story exercise unlocks after ~15+ words learned, appears every 3rd–4th session.

**Future extensibility:** Only `build_session()` changes. It receives `user_state`, `time`, `day` and returns a different ordered list of Exercise instances. The executor is generic — it iterates the list and knows nothing about exercise types.

## 3. Data Architecture

Three persistent JSON files per user, plus one static file.

### 3.1 vocab_state.json — Per-Word Learning State

Tracks every word the user has encountered.

```
{
  "<english_word>": {
    "ru":           string,      // Russian translation
    "topic":        string,      // Topic this word belongs to
    "isLearning":   boolean,     // true = still in active rotation
    "times_shown":  integer,     // Times shown as "learn this word"
    "times_tested": integer,     // Times shown as exercise question
    "results":      [boolean],   // Ordered history of correct/incorrect on exercises
    "last_seen":    ISO datetime // Last time word appeared in any context
  }
}
```

**Graduation rule:** If `results[-4:]` has 3+ `true` values → set `isLearning: false`.

**Re-entry rule:** If a graduated word is tested in review and answered wrong → set `isLearning: true`.

### 3.2 topics.json — Topic Registry

Tracks which topics exist and their progress.

```
{
  "<topic_name>": {
    "word_count": integer,    // Total words generated for this topic
    "started":    boolean     // Whether this topic has been introduced
  }
}
```

Initial topics (ordered): greetings, numbers & colors, family, food & restaurant, shopping, transport, doctor, travel, small talk, weather.

### 3.3 user_profile.json — Learner Profile

Combines LLM-generated summary with hard metrics.

```
{
  "summary":        string,    // LLM-generated 2-3 sentence learner description
  "metrics": {
    "words_learned":       integer,   // count of isLearning == false
    "words_in_progress":   integer,   // count of isLearning == true
    "sessions_completed":  integer,
    "total_time_minutes":  float,
    "avg_accuracy":        float,     // rolling average across all exercises
    "streak_days":         integer,
    "days_active":         integer
  },
  "weak_spots":     [string],   // e.g. ["articles", "plural forms"]
  "strong_topics":  [string],
  "current_topic":  string
}
```

**Summary refresh:** LLM regenerates `summary` every 5 sessions based on metrics, recent mistakes, and topics covered.

### 3.4 word_bank.json — Generated Word Repository (Static-ish)

Words generated by LLM, organized by topic. Acts as a buffer between LLM generation and active learning.

```
{
  "<topic_name>": [
    { "en": string, "ru": string }
  ]
}
```

## 4. Word Lifecycle

```
LLM generates word
       ↓
  word_bank.json  (stored, not yet active)
       ↓
  vocab_state.json  (isLearning: true, moved into active pool)
       ↓
  Shown in sessions → tested → results[] grows
       ↓
  3/4 correct → isLearning: false  (graduated)
       ↓
  Occasionally reviewed → wrong? → isLearning: true  (re-entered)
```

---

## 5. Word Generation (LLM)

Before each session, the system checks if the active pool needs refilling.

### 5.1 Trigger

`count(isLearning == true) < 10` → generate new words.

### 5.2 Topic Selection

Priority: pick the started topic with the lowest `word_count`. This keeps topics filling evenly rather than jumping between them.

If the lowest-count started topic has 30+ words → start the next unstarted topic.

If no topics are started → start the first one.

### 5.3 LLM Prompt

The prompt includes:

- Target topic
- User profile summary + metrics (from `user_profile.json`)
- Words already in this topic (from `word_bank.json`) to avoid duplicates
- Request for 5 new words as JSON array

The LLM returns `[{"en": "...", "ru": "..."}]`. These are appended to `word_bank.json` and `topics.json` word_count is incremented.

### 5.4 Pool Fill

After generation, words move from `word_bank` → `vocab_state` (with `isLearning: true`) until pool reaches target size (10).

## 6. VocabExercise — Detailed Flow

### Phase 1: Show (Learning)

- Pick 5 random words where `isLearning == true`
- Display each with translation: "🍎 apple — яблоко"
- Increment `times_shown` for each
- No user input required — pure exposure

### Phase 2: Test

- Pick 5 random words where `isLearning == true` (can overlap with Phase 1)
- Ask one by one: "Как по-английски 'яблоко'?"
- User types answer
- Evaluate: fuzzy string match against expected English word (tolerate minor typos)
- Record `true`/`false` in `results[]`, increment `times_tested`

### Phase 3: State Update

- Update `last_seen` for all touched words
- Check graduation: for each tested word, if `results[-4:].count(true) >= 3` → set `isLearning: false`
- Update `user_profile.json` metrics

### Answer Evaluation

MVP: case-insensitive fuzzy match (Levenshtein distance ≤ 1 for short words, ≤ 2 for longer). No LLM needed for this.

If wrong, show correct answer with encouragement: "Почти! Правильно: apple 🍎"

## 7. Session Execution Model

### 7.1 Exercise Base

Each exercise type is a class with two methods:

- `prompt(user_state)` → generates and returns the message to send
- `evaluate(user_message, user_state)` → grades response, returns feedback + state changes

Exercises are self-contained. They don't know about other exercises or their position in the session.

### 7.2 Session Builder

A pure function that returns an ordered list of exercise instances:

```
build_session(user_state, time?, day?) → [Exercise, Exercise, ...]
```

MVP always returns the same list. Future: branch on time/day.

### 7.3 Session Executor

A generic loop:

```
for each exercise in list:
    send exercise.prompt()
    wait for user response (with timeout)
    send exercise.evaluate(response)
save state
show session score
```

The executor handles timeout/skip logic. It does not know what kind of exercises it's running.

---

## 8. Skip & Absence Handling

- Session push sent → no response in 2 hours → mark session skipped
- Skipped session content merges into next session (1-2 extra review items in VocabExercise)
- Spaced repetition timer pauses on skip — words are not penalized
- Full day skipped → next session starts with review-only VocabExercise
- 2+ days idle → gentle nudge with one easy question (low friction re-engagement)
- Never stack more than 1 extra review block
- No guilt messages. Ever.

---

## 9. Scheduling

Cron job triggers 3 daily pushes (times configurable):

- Morning: ~9:00
- Afternoon: ~14:00
- Evening: ~20:00

Push message: simple prompt like "Готова к занятию? 🎯" or one quick word to spark engagement.

User replies → session starts. No reply → skip logic kicks in.

## 10. Tech Stack

| Component | Technology |
|---|---|
| Bot framework | OpenClaw (Telegram integration, LLM routing) |
| LLM | Claude API (via OpenClaw) |
| State storage | JSON files per user (MVP) |
| Scheduling | Cron on host VM |
| Voice/TTS | Telegram native voice messages |
| Fuzzy matching | Levenshtein distance (standard lib or small utility) |
| Hosting | Azure VM (existing OpenClaw instance) |

### Future upgrades (not MVP)

- JSON → SQLite when state gets complex or multi-user
- Analytics dashboard
- Configurable session pipelines per time/day
- Image-based exercises
- Conversation/roleplay exercises

## 12. Configuration Constants

| Constant | Value | Description |
|---|---|---|
| ACTIVE_POOL_SIZE | 10 | Target number of isLearning words |
| WORDS_PER_SHOW | 5 | Words shown in learning phase |
| WORDS_PER_TEST | 5 | Words tested in exercise phase |
| GRADUATION_THRESHOLD | 3 of last 4 | Correct answers to graduate |
| TOPIC_FULL_THRESHOLD | 30 | Words per topic before moving on |
| PROFILE_REFRESH_INTERVAL | 5 | Sessions between LLM summary refresh |
| SKIP_TIMEOUT_HOURS | 2 | Hours before marking session skipped |
| LLM_WORDS_PER_BATCH | 5 | Words requested from LLM per generation |
| SESSION_PUSH_TIMES | [9:00, 14:00, 20:00] | Daily push schedule |
